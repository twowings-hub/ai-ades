섹션 1 — 시스템 상태

6개 서비스의 실시간 헬스체크와 시스템 리소스 현황을 확인하는 화면입니다. 30초마다 자동으로 갱신됩니다.

서비스 헬스체크 표시

data-prep-agent:8010정상
modeling-agent:8011정상
execution-agent:8012정상
MLflow:5000정상
InfluxDB:8086지연 120ms
Grafana:3010정상
  • 상태 색상
    초록 — 정상 서비스가 응답하고 있습니다. 노랑 — 지연 응답이 느립니다(200ms 초과). 빨강 — 장애 응답 없음.
  • 응답시간 (ms)
    각 서비스의 /health 엔드포인트 응답 시간입니다. 정상 범위는 200ms 이하입니다. 200~500ms는 경고, 500ms 초과 또는 무응답은 장애로 판단합니다.

시스템 리소스 패널

  • CPU 사용률 (%)
    현재 호스트 CPU 사용률입니다. 모델 재학습 중에는 80~90%까지 올라갈 수 있습니다. 평상시 40% 이하가 정상입니다.
  • RAM 사용량 (GB)
    현재/전체 메모리 사용량입니다. Ollama 7b 모델 기준 약 6GB를 점유합니다. 총 32GB 중 25GB 이상 사용 시 경고 조치가 필요합니다.
  • Disk 사용량 (GB)
    호스트 디스크 사용량입니다. MLflow 아티팩트와 DB 백업 파일이 축적됩니다. 80% 이상 시 오래된 백업 파일을 정리하세요.

모델 성능 패널

  • depth R²
    Depth 예측 모델의 결정계수입니다. 1에 가까울수록 정확합니다. 0.75 이상 유지가 목표이며, 0.70 이하로 떨어지면 재학습이 필요합니다.
  • kerf R²
    Kerf 예측 모델의 결정계수입니다. depth R²와 함께 모델 품질을 나타냅니다.
  • OK F1
    품질 분류 모델에서 OK 클래스의 F1 점수입니다. 0.60 이상 유지가 목표입니다. OK 데이터가 부족(13.7%)하여 낮게 나올 수 있으며, 실험 데이터 누적 시 향상됩니다.
  • 마지막 학습일
    가장 최근 모델 학습이 완료된 날짜입니다. 신규 실험 데이터가 50건 이상 누적되면 재학습을 고려하세요.

LLM 상태 패널

  • Provider
    현재 사용 중인 LLM 공급자입니다. ollama(로컬), claude(API), openai(API) 중 하나입니다.
  • Model
    현재 로드된 모델명입니다. 예: qwen2.5:7b-instruct
  • 응답시간 avg
    LLM 설명 생성에 걸리는 평균 시간입니다. 7b 모델 기준 3~5초가 정상입니다. 10초 이상이면 Ollama 서비스를 재시작하세요.
  • 오늘 호출 횟수
    당일 /doe/suggest 호출 횟수입니다. API 모델(claude, openai) 사용 시 비용 계산에 활용하세요.

섹션 2 — 서비스 관리

각 컨테이너의 재시작과 로그를 관리하는 화면입니다.

서비스별 재시작 버튼

버튼동작주의사항
재시작해당 서비스 컨테이너를 재시작합니다. 약 10~30초 소요됩니다.재시작 중에는 해당 서비스의 API 호출이 실패합니다. 운영자에게 미리 알려주세요.
로그해당 서비스의 최근 로그 100줄을 팝업으로 표시합니다.오류 발생 시 로그에서 원인을 확인하세요.
execution-agent 재시작 제한: execution-agent는 자기 자신을 재시작할 수 없습니다. 버튼이 비활성화되어 있으며 클릭 시 터미널 명령어가 안내됩니다. 터미널에서 직접 실행하세요.

터미널에서 직접 재시작하는 방법

# 개별 서비스 재시작 docker-compose restart execution-agent docker-compose restart modeling-agent docker-compose restart data-prep-agent # 전체 서비스 재시작 (운영 중단 발생) docker-compose down docker-compose up -d # 로그 실시간 확인 docker-compose logs -f modeling-agent

전체 재시작

[전체 재시작] 버튼은 확인 모달이 표시된 후 실행됩니다. 모든 서비스가 동시에 재시작되므로 약 1~2분간 시스템이 중단됩니다. 운영 시간 중에는 사용을 피하세요.

섹션 3 — LLM 모델 선택

AI 자연어 설명 생성에 사용하는 LLM을 변경하는 화면입니다. 코드 수정이나 재시작 없이 즉시 전환됩니다.

전환 즉시 반영: 모델을 변경하고 [변경사항 적용]을 누르면 다음 /doe/suggest 호출부터 새 모델이 사용됩니다. 모든 변경은 감사 로그에 기록됩니다.

변경사항 적용.env 파일을 수정하고 llm_explainer를 런타임 재초기화합니다. 재시작 없이 즉시 적용됩니다.테스트 후에 적용하세요. 실패 시 Ollama 기본 모델로 자동 복구됩니다.

새 모델 설치 방법 (터미널)

# 사용 가능한 모델 목록 확인 (Admin Console에서도 조회 가능) ollama list # 새 모델 다운로드 (용량 주의: 14b ≈ 9GB, 32b ≈ 20GB) ollama pull qwen2.5:14b # 다운로드 완료 확인 후 Admin Console에서 전환 ollama list | grep qwen2.5:14b
중요: ollama pull로 다운로드를 완료한 후에 Admin Console에서 전환하세요. 모델이 설치되지 않은 상태에서 전환하면 LLM 설명 생성이 실패합니다 (나머지 API는 정상 작동).

섹션 4 — 모델 재학습

XGBoost 예측 모델을 새 데이터로 재학습하는 화면입니다.

현재 학습 데이터 현황

  • 총 실험 건수
    현재 experiments 테이블의 전체 데이터 건수입니다. 숫자가 증가할수록 AI 예측 정확도가 향상됩니다.
  • 마지막 학습일
    가장 최근에 모델 학습이 완료된 날짜입니다. 이 날짜 이후에 추가된 실험 데이터는 아직 반영되지 않았습니다.
  • 신규 미반영 건수
    마지막 학습 이후 새로 추가된 실험 건수입니다. 이 숫자가 50건 이상이면 재학습을 권장합니다.

자동 재학습 트리거 설정

  • 누적 건수 기준
    신규 데이터가 이 숫자 이상 누적되면 자동으로 재학습이 실행됩니다. 기본값은 50건입니다. 실험 빈도에 따라 조정하세요.
  • 자동 재학습 활성화
    체크박스로 자동 재학습 on/off를 설정합니다. 기본값은 활성화입니다.

수동 재학습 실행

섹션 5 — 판정 기준 설정

OK 판정 기준과 AI 탐색 공간을 변경하는 화면입니다. 변경 즉시 전 Agent에 반영되며 감사 로그에 기록됩니다.

최고 주의: 판정 기준 변경은 시스템 전체 동작에 영향을 미칩니다. 변경 전 반드시 고객사 품질 담당자와 협의하고, 변경 후 테스트를 통해 검증하세요.

Depth OK 범위

  • OK 최솟값 (초과)
    Depth가 이 값을 초과해야 OK 판정입니다. 기본값 0.0μm. 즉 Depth가 0이면 과가공(완전 관통)으로 판정됩니다.
  • OK 최댓값 (이하)
    Depth가 이 값 이하이어야 OK 판정입니다. 기본값 25.0μm. 고객사 품질 기준에 따라 조정합니다. 이 값을 낮추면 OK 조건이 더 엄격해집니다.
  • 이상값 플래그 기준
    Thickness가 이 값을 초과하는 실험 데이터에 이상값 플래그를 표시합니다. 기본값 140.0μm.

탐색 공간 설정

Bayesian Optimization이 탐색하는 파라미터 범위를 설정합니다.

  • Power 범위 (W)
    최솟값과 최댓값 사이에서 Power를 연속적으로 탐색합니다. 기본값: 2.8W ~ 59.8W. 소재 특성에 따라 범위를 좁히면 탐색 효율이 높아집니다.
  • Speed 선택지 (mm/s)
    쉼표로 구분된 이산값 목록입니다. 기본값: 200, 500, 1000. 설비 사양에 맞는 값만 포함하세요.
  • Defocus 선택지 (mm)
    쉼표로 구분된 이산값 목록입니다. 기본값: 0, 1, 2, 3, 4. 설비에서 지원하는 값만 포함하세요.

버튼 설명

버튼동작
변경사항 저장.env 파일과 런타임 설정을 동시에 업데이트합니다. 감사 로그에 이전값→새값이 기록됩니다.
기본값으로 초기화모든 설정을 시스템 기본값으로 되돌립니다. 초기화도 감사 로그에 기록됩니다.

섹션 8 — 감사 로그

모든 승인·거부·설정 변경·LLM 전환 이력을 확인하는 화면입니다. 최신순으로 표시되며 20건씩 페이지네이션됩니다.

로그 컬럼 설명

  • 발생 시각
    이벤트가 발생한 정확한 시각입니다. (예: 2025-06-11 14:32:05)
  • 담당자
    행동을 수행한 운영자 이름 또는 '시스템'(자동 실행 이벤트)입니다.
  • 유형 (action_type)
    approval 운영자 승인
    rejection 운영자 거부
    llm_change LLM 모델 전환
    setting_change 판정 기준 변경
    retrain 모델 재학습
    service_restart 서비스 재시작
  • 내용
    이벤트 상세 내용입니다. 설정 변경의 경우 "이전값 → 새값" 형태로 표시됩니다.

CSV 내보내기

[CSV 내보내기] 버튼으로 전체 감사 로그를 CSV 파일로 다운로드할 수 있습니다. 품질 감사나 문제 추적에 활용하세요.

섹션 9 — 데이터 관리

실험 데이터 백업, Excel 내보내기, 이상값 처리를 담당하는 화면입니다.

  • experiments
    전체 실험 데이터 건수입니다. 초기 POC 328건 + 실운영 누적 건수입니다. 이 숫자가 증가할수록 AI 정확도가 향상됩니다.

판정 기준을 잘못 변경했을 때 어떻게 원복하나요? ›
Admin Console → 판정 기준 설정 → [기본값으로 초기화] 버튼을 누르세요. 또는 직접 DEPTH_OK_MIN=0.0, DEPTH_OK_MAX=25.0으로 입력하고 저장하세요. 모든 변경 이력은 감사 로그에 남아 있습니다.
모델 성능이 갑자기 낮아졌을 때 어떻게 하나요? ›
MLflow에서 이전 학습 Run을 확인하고 성능이 좋았던 시점의 모델 파일(.pkl)을 /services/modeling-agent/models/에 복사하세요. 그 후 modeling-agent를 재시작하면 이전 모델로 복구됩니다.
실험 데이터를 실수로 삭제했을 때 어떻게 하나요? ›
Admin Console → 데이터 관리 → 백업 파일 목록에서 가장 최근 백업을 확인하세요. 터미널에서 psql로 백업 파일을 복구하면 됩니다. 백업이 매일 자동으로 실행되므로 최대 1일치 데이터 손실이 발생할 수 있습니다.
