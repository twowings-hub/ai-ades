0. 오늘의 목표 — 무엇을, 왜

RTX 5080(VRAM 16GB) GPU가 장착된 PC를 새로 구매해 nb-ai-ws로 명명했다. 앞으로의 구도는:

  • 개발은 계속 Surface 노트북에서 (코드 작성·UI 다듬기)
  • AI/LLM 같은 무거운 추론 작업은 nb-ai-ws의 GPU로 넘긴다(오프로드)
왜 오프로드가 필요한가 Surface의 GPU는 RTX 3050 Ti로 VRAM이 4GB뿐이다. 14B급 LLM은 4GB에 다 못 올라가 CPU로 흘러넘쳐(offload) 추론이 매우 느리다(수십 초~분, ReadTimeout 빈발). nb-ai-ws의 5080은 VRAM 16GB라 14B 모델을 통째로 GPU에 올려 빠르게 돌릴 수 있다. 그래서 "LLM은 nb-ai-ws, 개발은 Surface"로 역할을 나눈다.

오늘 목표는 사용자가 미리 해둔 셋업을 점검하고, 문제를 바로잡은 뒤 실제로 GPU LLM 오프로드가 동작하게 만드는 것이다.

1. 사전 점검 — 사용자가 해둔 셋업 검사

1-1환경설정 파일 구조 파악

프로젝트 루트에 환경변수 프로필이 두 개 새로 생겨 있었다.

파일내용의도
.env.surfaceDB·LLM·MLflow를 전부 로컬 컨테이너 이름(ollama, pm-postgres …)으로전부 Surface에서 실행
.env.nb-ai-ws전부 100.94.29.111(nb-ai-ws)로nb-ai-ws로 오프로드
.env실제로 적용되는 활성 파일위 둘 중 하나를 복사해서 사용
어떻게 확인했나 docker-compose.yml은 env_file: .env 와 변수치환에 .env를 쓴다. 즉 운영자는 머신에 맞는 프로필을 .env로 복사해 쓰는 구조. 활성 .env를 .env.surface와 diff 해보니 .env = .env.nb-ai-ws(오프로드 프로필)와 동일했다.

1-2발견한 문제들

심각도문제설명
치명MODELING_AGENT_URL 줄 깨짐.env에 키 없이 http://100.94.29.111:8011 한 줄만 있어, dotenv가 무시 → 기본값(로컬)로 폴백. 모델 학습/예측 오프로드가 조용히 안 됨.
보안API 키 평문 노출 위험.env.surface/.env.nb-ai-ws에 실제 Claude·OpenAI 키가 있는데 .gitignore가 .env만 무시 → git add . 한 번에 푸시될 위험.
주의컨테이너가 옛 env로 동작.env는 편집됐지만 컨테이너는 생성 시점 env로 계속 돌아감(env_file은 런타임 재읽기 안 함).
치명nb-ai-ws Ollama 미기동11434 포트 연결 거부. 오프로드의 핵심인 GPU LLM이 떠 있지 않음.
데이터데이터는 로컬에만, nb-ai-ws DB는 빔아래 1-3에서 상세.

1-3가장 중요한 발견: 데이터가 어디 있는가

오프로드 프로필은 DB까지 nb-ai-ws로 가리키고 있었다. 그런데 실제 데이터 위치를 비교해보니:

저장소Surface (로컬)nb-ai-ws
Postgres ades_dbexperiments 350 · recipes 23 · approvals 47스키마 13테이블만, 데이터 0건
MLflowai-ades-laser-cutting 학습 이력 보유Default만 (빔)
학습된 모델로컬 modeling-agent에 존재없음 → /predict 실패 위험
어떻게 확인했나 실행 중인 execution-agent 컨테이너 안에서 양쪽 Postgres에 직접 접속해 테이블 행수를 셌고(psycopg2), MLflow는 /api/2.0/mlflow/experiments/search를 양쪽에 호출해 비교했다.
위험 이 상태로 오프로드 프로필을 그대로 적용(컨테이너 재생성)하면, 앱이 실험 0건·레시피 0건·모델 없음인 빈 백엔드를 바라보게 된다. 데이터가 삭제되는 건 아니지만(로컬 볼륨에 남음) 앱에서 안 보이고 예측이 깨진다. 그래서 진행을 멈추고 방향을 다시 정했다.

2. 핵심 발견과 방향 결정 — "LLM만 오프로드"

"무거운 작업을 nb-ai-ws로"라는 큰 방향은 맞지만, 데이터까지 통째로 옮기는 건 별도의 마이그레이션 작업이고 오늘 안 된 상태였다. 두 가지 선택지를 비교했다.

선택지내용판단
A. LLM만 오프로드DB·MLflow·모델·데이터는 Surface 로컬 단일 소스로 유지, nb-ai-ws는 Ollama(GPU LLM)만 담당채택 데이터 안전, 즉시 가능, 되돌리기 쉬움
B. 전체 마이그레이션로컬 데이터를 nb-ai-ws로 덤프·복원 후 단일 백엔드로 전환나중에 별도 진행 (이관 스크립트 필요)
왜 A인가 오프로드가 진짜 노리는 이득(=GPU로 LLM 빠르게)은 Ollama만 nb-ai-ws로 보내면 100% 달성된다. DB·MLflow는 무거운 연산이 아니라 옮길 이유가 약하고, 옮기면 데이터 이원화(split-brain) 위험만 생긴다. 그래서 "Ollama만 nb-ai-ws, 나머지는 로컬"이 가장 합리적.

3. Surface 측 설정 정리

3-1보안: 키 파일을 git 추적에서 제외

어떻게 .gitignore에 패턴 추가 (예시 파일은 계속 추적)
# 환경변수 (실제 키 포함 — .env.example만 추적)
.env
.env.*
!.env.example

※ 다행히 이전에 커밋된 적은 없어(추적 파일은 .env.example뿐) 유출은 없었다.

3-2깨진 MODELING_AGENT_URL 줄 복구

# Before (키가 없어 무시됨)
#MODELING_AGENT_URL=http://modeling-agent:8011
http://100.94.29.111:8011
# After
MODELING_AGENT_URL=http://100.94.29.111:8011

전체 마이그레이션을 대비한 .env.nb-ai-ws 프로필의 정합성을 위해 수정.

3-3.env를 "로컬 + Ollama만 nb-ai-ws"로 구성

어떻게 .env.surface(전부 로컬)를 .env로 복사한 뒤, Ollama 주소 한 줄만 nb-ai-ws로 변경.
cp .env.surface .env
# .env  ← Ollama만 nb-ai-ws, 나머지는 전부 로컬
LLM_PROVIDER=claude
#OLLAMA_BASE_URL=http://ollama:11434
OLLAMA_BASE_URL=http://100.94.29.111:11434   ← 이 줄만 변경
POSTGRES_HOST=pm-postgres        # 로컬 유지
MLFLOW_TRACKING_URI=http://mlflow:5000   # 로컬 유지
MODELING_AGENT_URL=http://modeling-agent:8011  # 로컬 유지

3-4execution-agent 재생성으로 새 설정 반영

왜 재생성인가 env_file은 컨테이너를 만드는 시점에만 읽힌다. .env를 고쳐도 이미 떠 있는 컨테이너엔 반영 안 됨 → --force-recreate 필요. Ollama 주소를 쓰는 건 execution-agent 하나뿐이라 그것만 재생성.
docker-compose --profile agents up -d --force-recreate execution-agent
검증 재생성 후 컨테이너 내부 env가 OLLAMA_BASE_URL=…100.94.29.111…, 나머지는 로컬로 정확히 반영됐고, /health 정상, 데이터도 experiments 350건 그대로 보였다.

4. nb-ai-ws 원격 접속 분투기 (PSRemoting → RDP)

이제 nb-ai-ws에 Ollama를 띄울 차례. 여기서 Windows 원격 접속 특유의 함정을 여러 개 만났다. 단계별로 왜 막혔고 어떻게 풀었는지 기록한다.

4-1현황 확인 → Ollama가 아예 없음

nb-ai-ws에서 docker ps 결과: data-prep·modeling·postgres·influxdb·mlflow·grafana·kafka는 떠 있는데 Ollama 컨테이너가 없었다. 11434 거부의 원인은 "미기동"이 아니라 "미설치"였다.

4-2난관 1 Docker 자격증명 에러

docker run ... ollama/ollama
→ error getting credentials - err: exit status 1,
  out: `A specified logon session does not exist. It may already have been terminated.`
왜 났나 nb-ai-ws에 PowerShell 원격(PSRemoting/WinRM) 세션으로 들어가 있었다(프롬프트가 [100.94.29.111]: PS> 형태). 이런 네트워크 로그온 세션에서는 Docker Desktop의 자격증명 헬퍼가 Windows 자격증명 볼트에 접근하지 못한다. ollama/ollama는 공개 이미지라 로그인이 필요 없는데도 헬퍼가 끼어들어 막혔다.
시도한 우회들 (모두 실패) ~/.docker/config.json의 credsStore 비우기, 빈 {} 작성, docker --config로 빈 설정 강제 지정 — 전부 같은 에러. 결론: 설정 문제가 아니라 원격 세션의 구조적 한계. → 인터랙티브 RDP 세션이 필요.

4-3난관 2 RDP가 꺼져 있어 연결 불가

mstsc로 붙으니 "원격 컴퓨터가 꺼져 있음" 류 에러. RDP가 비활성 상태였다. 이미 잘 되는 PSRemoting 세션을 이용해 RDP를 켰다(레지스트리·방화벽 명령은 자격증명 볼트가 필요 없어 원격 세션에서도 동작).

Set-ItemProperty -Path 'HKLM:\System\CurrentControlSet\Control\Terminal Server' -Name fDenyTSConnections -Value 0
Enable-NetFirewallRule -DisplayGroup "원격 데스크톱"   # 영문판: "Remote Desktop"
Set-ItemProperty -Path 'HKLM:\...\WinStations\RDP-Tcp' -Name UserAuthentication -Value 1  # NLA(선택)

4-4난관 3 RDP 자격증명 — 로컬 계정 지정

mstsc가 Surface의 마이크로소프트 계정으로 붙으려 해 "자격 증명이 작동하지 않습니다". nb-ai-ws의 로컬 계정으로 강제해야 했다.

어떻게 "다른 계정 사용" → 사용자명 .\ades (맨 앞 .\ = 로컬 머신 계정) + ades의 윈도우 비밀번호.

4-5첫 로그인 설정 화면 → 그냥 통과

RDP 첫 인터랙티브 로그인이라 "윈도우 설치처럼 보이는" 첫 로그인 설정(개인정보/지역/PIN) 화면이 떴다. OS 재설치가 아님(Docker가 20시간째 가동 중이었음). 기본값으로 클릭해 통과하니 바탕화면 도착.

분기 만약 "Windows 설치 / 드라이브 선택 / 파티션" 문구가 보였다면 즉시 멈춰야 한다(재설치=데이터 손실). 이번 건은 단순 첫 로그인 설정이라 안전하게 진행.

4-6난관 4 docker 파이프 permission denied

permission denied while trying to connect to the docker API at npipe:////./pipe/docker_engine
왜 RDP 바탕화면에서 연 PowerShell이 일반 권한이라 Docker 엔진 파이프 접근 불가(PSRemoting은 관리자 토큰이라 됐던 것).
어떻게 PowerShell을 관리자 권한으로 실행(우클릭 → 관리자 권한으로 실행). 상시 해결책: net localgroup docker-users ades /add 후 재로그인.

5. GPU Ollama 기동 + 모델 적재

5-1Ollama 컨테이너 기동 (GPU 패스스루)

docker run -d --gpus=all -v ollama:/root/.ollama -p 11434:11434 --name ollama ollama/ollama
docker exec ollama nvidia-smi
옵션 의미 --gpus=all GPU를 컨테이너에 노출(필수). · -v ollama:/root/.ollama 모델을 볼륨에 영구 저장(컨테이너 지워도 모델 유지). · -p 11434:11434 호스트 0.0.0.0에 바인딩 → Surface가 Tailscale로 바로 접속(별도 OLLAMA_HOST 설정 불필요).
성공 nvidia-smi에 NVIDIA GeForce RTX 5080 · 16303MiB 표시 → GPU 패스스루 정상.

5-2모델 다운로드 — 14B@Q6 (16GB의 sweet spot)

docker exec ollama ollama pull qwen2.5:14b-instruct-q6_K   # 약 12GB
docker exec ollama ollama list
왜 14B-Q6인가 VRAM 16GB에서 14B를 Q6_K(~12GB)로 올리면 모델 전체가 GPU에 들어가고, KV캐시·채팅 매뉴얼 in-context 주입에 쓸 여유(약 4GB)도 남는다. 32B는 강한 양자화(Q3)로만 들어가 오히려 14B-Q6보다 품질이 떨어질 수 있어 피했다. 7B에서 한 단계 품질을 올린 선택.
메모 호스트엔 ollama 바이너리가 없고 컨테이너 안에 있다. 항상 docker exec ollama ollama … 로 실행할 것.

6. Surface↔nb-ai-ws end-to-end 검증

6-1연결 확인 (Surface에서)

curl http://100.94.29.111:11434/api/tags
→ HTTP 200 (connect 4ms)

아까 거부되던 11434가 이제 Surface에서 정상 응답. Tailscale 경로 + 0.0.0.0 바인딩이 잘 맞물렸다.

6-2실제 추론 1회 (전체 경로 관통)

Surface에서 /api/generate를 호출해 nb-ai-ws GPU로 실제 추론을 돌렸다.

※ 테스트 시 응답이 깨진 글자로 보인 건 Surface 쉘이 한글 프롬프트를 잘못 인코딩해 보낸 탓이며, 실제 앱(Python·UTF-8)에서는 정상.

7. LLM 프로바이더 전환 (Claude → Ollama)

7-1관리자 API로 런타임 전환

왜 API/콘솔로 전환하나 execution-agent에는 재시작 없이 LLM을 바꾸는 엔드포인트(/admin/llm/switch)가 있다. 런타임으로 즉시 적용되고 .env에도 기록(update_env)되어 영구 반영된다.
POST /admin/llm/switch
{ "provider": "ollama", "model": "qwen2.5:14b-instruct-q6_K" }
→ current: { provider: ollama, model: qwen2.5:14b-instruct-q6_K }
검증 /admin/llm/test 실연결 테스트 성공 · 473ms(웜 상태). .env도 LLM_PROVIDER=ollama로 갱신됨.

7-2AI 채팅 프로바이더는 보류

이 콘솔 전환은 "AI 설명"용 LLM_PROVIDER만 바꾼다. "AI 채팅"용 CHAT_LLM_PROVIDER는 별도 값이라 아직 claude로 둔 상태(전환하려면 .env 수정 + execution-agent 재생성 필요). 채팅은 자유 대화라 품질 차이가 더 느껴질 수 있어 일부러 분리해둔 구조.

남은 일 (선택)

  • UI에서 직접 테스트 — http://localhost:5173 접속 → /admin(LLM 관리·연결 테스트) → /(예측 후 AI 설명 생성 확인). 첫 설명은 콜드스타트로 느릴 수 있음.
  • AI 채팅도 Ollama로 할지 결정 — 비용 0·폐쇄망 일관성 vs Claude의 대화 품질. 전환 시 CHAT_LLM_PROVIDER=ollama + execution-agent 재생성.
  • (먼 훗날) 전체 마이그레이션 — 데이터까지 nb-ai-ws로 옮길 경우 Postgres/InfluxDB/MLflow/모델 덤프·복원 후 .env.nb-ai-ws 프로필로 전환.
  • nb-ai-ws 정리(선택) — nb-ai-ws가 안 쓰는 ades 스택을 한 벌 더 돌리고 있음. GPU 박스를 가볍게 두려면 Ollama만 남기고 내려도 됨.

운영 팁 (재발 방지)

  • nb-ai-ws에서 docker 명령은 RDP + 관리자 권한 PowerShell에서. PSRemoting([100.94.29.111]: 프롬프트) 세션에선 docker pull이 자격증명 에러로 막힌다.
  • 호스트엔 ollama가 없고 컨테이너 안에 있으니 항상 docker exec ollama ollama ….
  • .env 편집 후엔 반드시 --force-recreate로 컨테이너 재생성해야 반영된다.
  • RDP가 막히면 PSRemoting 세션으로 fDenyTSConnections=0 + 방화벽 규칙으로 켤 수 있다. 로그인 사용자명은 로컬 계정이라 .\ades.
